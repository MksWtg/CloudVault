
Domain driven design is a design practice when writing software to orient it around the domain that the software operates in- real objects and the rules of the real world/domain.

DDD suggests you should use the same language whether you are a developer or a domain expert. Domain experts and product people should be able to read the code and figure out what is going on. DDD prioritizes expressing how the business actually works in the code.

If you have written OO code before, you might be thinking... this sounds like OO! OO suggests you to model real world things as objects, and obviously to name the classes after their real world counterparts, and to model real world processes as methods. 

They are different things.

First thing to acknowledge is that OO is not actually just about making real world things objects. This might have been how it started and how it got its name, but think about the pillars of OO- encapsulation, inheritance, polymorphism, abstraction- these are core OO. Or the SOLID principals ([[SOLID]]). OO is all of these things too. It is a concrete technology.

DDD refers to just the first part- making real world things into objects.

This means you can write DDD code in functional languages like F# or scala. Or you can write it with C# and make your DDD app via OO.

Also: DDD came like 20 years after OO had been in industyr.
