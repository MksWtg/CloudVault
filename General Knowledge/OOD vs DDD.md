
Domain driven design is a design practice when writing software to orient it around the domain that the software operates in- real objects and the rules of the real world/domain.

DDD suggests you should use the same language whether you are a developer or a domain expert. Domain experts and product people should be able to read the code and figure out what is going on. DDD prioritizes expressing how the business actually works in the code.

If you have written OO code before, you might be thinking... this sounds like OO! OO suggests you to model real world things as objects, and obviously to name the classes after their real world counterparts
